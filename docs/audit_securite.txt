# AUDIT DE SÉCURITÉ - TATTOOLIB SAAS

## 1. Analyse des Policies (Gestion Autorisations)

### ✅ Points Forts
- **BookingRequestPolicy**: Contrôle d'accès granulaire par rôle et statut
- **ConversationPolicy**: Vérification participants avec `hasParticipant()`
- **Gate usage**: Utilisation cohérente dans contrôleurs API

### ⚠️ Vulnérabilités Identifiées

#### BookingRequestPolicy.php
**Ligne 16**: `$bookingRequest->client->id` - Possible null exception si client non trouvé
**Ligne 20**: `$bookingRequest->bookable->user_id` - Risque null si bookable undefined

#### ConversationPolicy.php
**Ligne 41**: `hasRole('admin')` - Vérification simple, pas de cache ou rate limiting

## 2. Analyse Middlewares

### ✅ Bonnes Pratiques
- **EnsureSanctumAuthentication**: Hash SHA256 tokens, vérification expiry
- **EnsureUserHasRole**: Validation rôle avec redirection login
- **EnsureUserIsAdmin**: Gestion AJAX/Livewire différenciée

### 🚨 Failles Critiques

#### EnsureSanctumAuthentication.php
**Ligne 20-22**: 
```php
$accessToken = PersonalAccessToken::where('token', hash('sha256', $token))
    ->where('expires_at', '>', now())
    ->first();
```
**Risque**: Requête directe sans protection contre timing attacks

#### EnsureUserIsAdmin.php
**Ligne 28-32**: Logique complexe avec conditions multiples - risque de bypass

## 3. Authentification

### LoginController.php
**Ligne 19**: `Auth::attempt($credentials)` - ✅ Utilisation native Laravel
**Ligne 20**: `session()->regenerate()` - ✅ Protection fixation session

### AuthController.php (API)
**Ligne 39-44**: 
```php
if (!Auth::attempt($request->only('email', 'password'))) {
    throw ValidationException::withMessages([
        'email' => ['Les identifiants fournis sont incorrects.'],
    ]);
}
```
**Risque**: Message d'erreur générique pourrait révéler existence email

## 4. Contrôleurs API - Failles Majeures

### BookingRequestController.php

#### 🚨 Injection SQL Potentielle
**Ligne 188**: 
```php
$slots = Availability::getAvailableSlotsForDay(
    $bookingRequest->bookable->id,  // Pas de validation
    $validated['scheduled_date']
);
```

#### 🚨 Mass Assignment
**Ligne 194-204**: Update direct avec données utilisateur non filtrées
```php
$bookingRequest->update([
    'status' => BookingRequest::STATUS_ACCEPTED,
    'accepted_at' => now(),
    // ... données utilisateur directes
]);
```

#### 🚨 IDOR (Insecure Direct Object Reference)
**Ligne 246**: 
```php
if (!$user->isClient() || $bookingRequest->client_id !== $user->client->id) {
```
**Risque**: Comparaison directe ID sans vérification supplémentaire

### PaymentController.php

#### 🚨 Logique Paiement Critique
**Ligne 28-30**: 
```php
if ($request->user()->client->id !== $bookingRequest->client_id) {
    return response()->json(['error' => 'Unauthorized'], 403);
}
```
**Risque**: Pas de vérification montant minimum, pas de rate limiting

#### ⚠️ Stripe Configuration
**Ligne 17**: `Stripe::setApiKey(config('services.stripe.secret'));`
**Risque**: Clé API en config, pas de rotation automatique

### MessageController.php

#### 🚨 File Upload Vulnerabilities
**Ligne 67-73**: 
```php
if ($request->hasFile('attachment')) {
    $file = $request->file('attachment');
    $attachmentType = $this->determineAttachmentType($file);
    $message->addAttachment($file, $attachmentType);
}
```
**Risques**:
- Pas de validation taille fichier
- Pas de validation type MIME côté serveur
- Pas de scan antivirus
- Stockage direct sans sandbox

#### 🚨 Path Traversal
**Ligne 159**: 
```php
return response()->download($media->getPath(), $media->file_name);
```
**Risque**: `getPath()` pourrait retourner chemin système

## 5. Validation des Entrées

### ✅ Bonnes Pratiques
- Utilisation `validate()` dans AuthController
- Form Requests pour BookingRequest

### ⚠️ Faiblesses
- **MessageController**: Pas de validation sur `content` (XSS possible)
- **PaymentController**: Pas de validation montant maximum
- **File uploads**: Validation MIME uniquement côté client

## 6. Gestion Sessions & Tokens

### ✅ Sécurités
- Sanctum avec expiration tokens
- Regeneration session login
- CSRF protection Laravel native

### ⚠️ Améliorations Nécessaires
- Pas de rate limiting login
- Tokens non révoqués automatiquement
- Pas de détection d'anomalies connexion

## 7. Protection Routes Sensibles

### ✅ Middleware Appropriés
- `auth:sanctum` sur routes API
- Vérifications rôle dans contrôleurs
- Gates policies sur modèles

### 🚨 Routes Critiques Non Protégées
- `/api/bookings/{bookingRequest}/payment/deposit` - Pas de vérification montant
- `/api/messages/{message}/download` - Download direct fichiers
- Webhooks Stripe sans signature verification

## 8. Vulnérabilités Spécifiques

### XSS (Cross-Site Scripting)
**Risque Élevé**:
- Messages utilisateur non échappés dans MessageController
- Contenu conversation stocké sans validation
- Pas de CSP headers

### CSRF (Cross-Site Request Forgery)
**Risque Moyen**:
- API avec tokens Sanctum (protégé)
- Web forms Laravel (protégé)
- Webhooks sans vérification signature

### SQL Injection
**Risque Faible**:
- Utilisation Eloquent (protégé)
- Quelques requêtes brutes nécessitant validation

### File Upload
**Risque Critique**:
- Upload multiples sans restrictions
- Pas de validation taille/type serveur
- Stockage direct accessible

## 9. Recommandations Prioritaires

### 🚨 Urgent (Critique)
1. **Sécuriser uploads**: Validation taille, type, scan antivirus
2. **Rate limiting**: Login, paiements, uploads
3. **Input validation**: Échapper contenu utilisateur
4. **Webhook security**: Vérifier signature Stripe

### ⚠️ Important (Élevé)
1. **Rate limiting API**: Limiter requêtes par utilisateur
2. **Montant maximum**: Plafonner paiements
3. **Audit logs**: Tracer actions sensibles
4. **Headers sécurité**: CSP, HSTS, X-Frame

### 📝 Améliorations (Moyen)
1. **2FA obligatoire**: Admins et paiements
2. **Token rotation**: Renouvellement automatique
3. **Détection anomalies**: Connexions suspectes
4. **Encryption**: Données sensibles chiffrées

## 10. Tests de Pénétration Suggérés

### Scénarios Critiques
1. **Upload malveillant**: Tenter上传 executable
2. **XSS stored**: Injecter script dans message
3. **Payment bypass**: Manipuler montant acompte
4. **IDOR attack**: Accéder données autres utilisateurs
5. **Rate limit test**: Flood login/paiements

## 11. Score de Sécurité Global

**Note: 6/10**

- ✅ **Authentification**: 8/10 (Laravel robuste)
- ⚠️ **Autorisations**: 7/10 (Policies bien structurées)
- 🚨 **Validation**: 4/10 (Faiblesses uploads/XSS)
- ⚠️ **Paiements**: 5/10 (Stripe bien intégré mais validations manquantes)
- 🚨 **File handling**: 3/10 (Vulnérabilités critiques)

## Conclusion

L'architecture de base est solide avec Laravel, mais des vulnérabilités critiques nécessitent une action immédiate, particulièrement sur les uploads de fichiers et la validation des entrées utilisateur. La priorité est de sécuriser les fonctionnalités de paiement et de messagerie avant la mise en production.
