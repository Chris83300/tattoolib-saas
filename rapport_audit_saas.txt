==============================================
## 1. STRUCTURE & ARCHITECTURE
==============================================

### Vue générale de l'architecture
- **Framework** : Laravel 12.0 avec PHP 8.2+
- **Frontend** : Livewire 3.7 + Vue.js (Vite) + TailwindCSS 4.0
- **Base de données** : SQLite (développement) / MySQL/MariaDB/PostgreSQL (production)
- **Authentification** : Laravel Fortify + Sanctum (API)
- **Gestion médias** : Spatie MediaLibrary
- **Permissions** : Spatie Laravel Permission
- **Paiements** : Stripe Cashier + Stripe Connect
- **Notifications** : Firebase Cloud Messaging (FCM)

### Architecture multi-rôles
Le projet utilise une architecture polymorphique basée sur les rôles :
- **User** : modèle principal avec champ `role`
- **Profils spécialisés** : Client, Tattooer, Pierceur, Studio, StudioArtist
- **Relation polymorphique** : User -> Profile selon le rôle

### Points d'entrée principaux
- **Web** : routes/web.php (343 lignes) - Routes traditionnelles + Livewire
- **API** : routes/api.php (280 lignes) - API RESTful avec authentification Sanctum
- **Admin** : Filament panels pour l'administration

### Flux de données principales
1. **Inscription** : User -> Profile spécifique -> Vérification
2. **Booking Request** : Client -> Tattooer/Pierceur -> Acceptation -> Paiement -> RDV
3. **Messagerie** : Conversation -> Messages -> Notifications FCM
4. **Portfolio** : Upload médias -> Spatie MediaLibrary -> Affichage public

### Dépendances principales
- Laravel 12.0 (framework core)
- Livewire 3.7 (composants réactifs)
- Filament 4.5 (admin panel)
- Stripe Cashier 16.2 (gestion abonnements)
- Spatie MediaLibrary 11.17 (gestion fichiers)
- Spatie Permission 6.24 (gestion permissions)

## 3. PERFORMANCES
==============================================

### Problèmes de performance identifiés

#### Requêtes N+1
**Fichier : TattooerController.php (lignes 312-326, 333-347)**
```php
foreach ($appointments as $appointment) {
    $events[] = [
        'title' => 'RDV - ' . $appointment->client->first_name . ' ' . $appointment->client->last_name,
        // ...
    ];
}
```
**Problème** : `$appointment->client` génère N+1 requêtes malgré le `with('client')`

**Fichier : TattooerController.php (lignes 364-372)**
```php
->withCount(['messages as unread_count' => function($q) {
    $q->where('sender_type', 'client')
      ->whereDoesntHave('conversation.participants', function($subQ) {
          $subQ->where('user_id', auth()->id())
                ->whereNotNull('last_read_at');
      });
}])
```
**Problème** : Sous-requête complexe dans chaque itération

#### Requêtes non optimisées
**Fichier : TattooerController.php (lignes 70-98)**
```php
$stats = [
    'pending_requests' => BookingRequest::where('bookable_id', $tattooer->id)
        ->where('bookable_type', 'App\Models\Tattooer')
        ->where('status', 'pending')
        ->count(),
    // ... 5 autres requêtes similaires
];
```
**Problème** : 6 requêtes séparées pour les stats du dashboard

#### Chargement de données inutiles
**Fichier : TattooerController.php (lignes 150-160)**
```php
$counts = [
    'pending'   => BookingRequest::where('bookable_type', 'App\\Models\\Tattooer')->where('bookable_id', $tattooer->id)->where('status', 'pending')->count(),
    'accepted'  => BookingRequest::where('bookable_type', 'App\\Models\\Tattooer')->where('bookable_id', $tattooer->id)->whereIn('status', [...])->count(),
    // ... 3 autres requêtes similaires
];
```
**Problème** : Double comptage (déjà fait dans $requests)

### Optimisations recommandées
1. **Utiliser eager loading** : `with()` pour éviter N+1
2. **Compter en une seule requête** : `countBy('status')` ou `groupBy()`
3. **Mettre en cache les stats** : Dashboard stats changent peu
4. **Pagination** : Pour les listes de conversations/messages

## 4. SÉCURITÉ
==============================================

### Forces
- **Hashage mots de passe** : Hash::make() correctement utilisé
- **Validation des entrées** : validate() systématique
- **Authentification robuste** : Fortify + Sanctum
- **Protection CSRF** : Laravel middleware activé

### Vulnérabilités potentielles

#### Validation insuffisante
**Fichier : TattooerController.php (lignes 423-426)**
```php
$validated = $request->validate([
    'content' => 'required_without:attachments|string|max:2000',
    'attachments.*' => 'nullable|file|mimes:jpeg,jpg,png,webp,pdf|max:10240',
]);
```
**Problème** : Pas de validation XSS pour le contenu, extension .pdf autorisée (risque)

#### Autorisation basique
**Fichier : TattooerController.php (lignes 383-384)**
```php
abort_unless($bookingRequest->bookable_id == auth()->user()->tattooer->id &&
            $bookingRequest->bookable_type == 'App\Models\Tattooer', 403);
```
**Problème** : Vérification manuelle, pas de Gates/Policies

#### Exposition de données sensibles
**Fichier : routes/api.php (lignes 68-70)**
```php
Route::get('/user', function (Request $request) {
    return $request->user()->load(['client', 'tattooer']);
});
```
**Problème** : Expose toutes les données utilisateur sans filtrage

#### Upload de fichiers
**Fichier : TattooerController.php (lignes 459-463)**
```php
if ($request->hasFile('attachments')) {
    foreach ($request->file('attachments') as $file) {
        $message->addMedia($file)->toMediaCollection('attachments');
    }
}
```
**Problème** : Pas de validation du contenu réel du fichier (MIME spoofing)

#### Manque de rate limiting
- **API routes** : Pas de throttling visible
- **Upload** : Pas de limite de taille globale par utilisateur

### Recommandations sécurité
1. **Implémenter des Policies** : Pour autorisations fines
2. **Ajouter rate limiting** : Sur API et uploads
3. **Sanitizer inputs** : Pour prévenir XSS
4. **Scanner uploads** : Vérification réelle des fichiers
5. **Logging sécurité** : Tentatives d'accès non autorisées
