==============================================
## 1. VUE D'ENSEMBLE DE L'ARCHITECTURE
==============================================

### Type d'application
**SaaS Multi-rôles** pour la gestion de rendez-vous de tattoo/piercing

### Stack Technique
- **Backend** : Laravel 12.0 + PHP 8.2+
- **Frontend** : Livewire 3.7 + Vue.js + TailwindCSS 4.0
- **Base de données** : SQLite (dev) / MySQL/PostgreSQL (prod)
- **Authentification** : Laravel Fortify + Sanctum (API)
- **Médias** : Spatie MediaLibrary
- **Permissions** : Spatie Laravel Permission
- **Paiements** : Stripe Cashier + Stripe Connect
- **Notifications** : Firebase Cloud Messaging

### Architecture Pattern
- **MVC classique** avec polymorphisme avancé
- **Multi-tenancy implicite** via rôles
- **Domain-driven design** partiel (traits pour logique métier)

## 2. ENTITÉS PRINCIPALES ET RESPONSABILITÉS
==============================================

### 🏗️ Entités Core (Foundation)

#### User (Modèle principal)
**Responsabilités** :
- Authentification et autorisation de base
- Point d'entrée polymorphique vers les profils
- Gestion des tokens API et notifications

**Champs clés** :
- `role` : client, tattooer, pierceur, studio, studio_artist, admin
- `status` : active, pending_verification, suspended
- `is_admin`, `is_studio_owner`, `is_studio_artist`

**Relations** :
- `profile()` : polymorphique vers le profil spécialisé
- Relations directes : `client()`, `tattooer()`, `pierceur()`, `studio()`, `studioArtist()`

---

### 👥 Entités Client-Side

#### Client (Profil client)
**Responsabilités** :
- Gestion des informations personnelles client
- Historique des tattoos et rendez-vous
- Gestion du statut (blacklist, no-show)
- Consentements et fiches de suivi

**Champs clés** :
- `first_name`, `last_name`, `phone`, `birth_date`
- `no_show_count`, `is_blacklisted`, `blacklist_reason`

**Relations** :
- `user()` :belongsTo User
- `bookingRequests()` :hasMany BookingRequest
- `appointments()` :hasMany Appointment
- `tattooHistory()` :hasMany TattooHistory
- `consent()` :hasOne Consent

**Logique métier** :
- Auto-blacklist après 3 no-shows
- Vérification majorité légale
- Calcul statut (nouveau, fidèle, blacklisté)

---

### 🎨 Entités Artist-Side

#### Tattooer (Artiste tattoo)
**Responsabilités** :
- Profil professionnel et portfolio
- Gestion des disponibilités et tarifs
- Conformité réglementaire et Stripe Connect
- Workflow de réservation

**Traits utilisés** :
- `HasSubscription` : Gestion abonnements FREE/PRO

**Champs clés** :

**Relations** :
- `user()` :belongsTo User

- `appointments()` :morphMany Appointment
- `availabilities()` :morphMany Availability

**Collections médias** :
- `avatar` : Photo profil (unique)

**Relations** :

### 🔄 Entités Workflow

#### BookingRequest (Demande de réservation)
**Responsabilités** :
- Centraliser le workflow de réservation
- Gérer les états et délais
- Calculer les acomptes et tarifs

**Champs principaux** :
- `bookable_id`, `bookable_type` : Polymorphique (Tattooer/Pierceur/StudioArtist)
- `status` : pending, accepted, awaiting_deposit, deposit_paid, design_sent, confirmed, completed, cancelled
- `total_deposit_amount`, `total_price` : Financier
- `proposed_dates` : Dates proposées par l'artiste
- `chat_status`, `chat_closes_at` : Gestion chat temporaire

**Relations** :
- `client()` :belongsTo Client
- `bookable()` :morphTo (artiste)
- `conversation()` :hasOne Conversation
- `messages()` :hasMany Message
- `appointment()` :hasOne Appointment

**Workflow complet** :
1. **pending** : Client soumet demande
2. **accepted** : Artiste accepte avec modal (prix, dates, acompte)
3. **awaiting_deposit** : En attente paiement acompte
4. **deposit_paid** : Acompte payé, chat activé
5. **design_sent** : Design envoyé, validation client
6. **confirmed** : RDV confirmé, création appointment
7. **completed** : RDV terminé, création tattoo_history

---

#### Conversation (Chat temporaire)
**Responsabilités** :
- Gérer le cycle de vie du chat
- Expiration automatique selon plan
- Préservation des images importantes

**Cycle de vie** :
1. **deposit_pending** : Chat actif jusqu'à paiement acompte
2. **permanent** : Chat permanent après acompte
3. **post_appointment** : Post-RDV, suppression selon plan
4. **archived** : Archivage permanent (plan PRO)

**Champs clés** :
- `expiry_type` : Type d'expiration
- `expires_at` : Date d'expiration
- `deposit_deadline_at` : Délai acompte
- `images_preserved` : Images sauvegardées

**Relations** :
- `bookingRequest()` :belongsTo BookingRequest
- `messages()` :hasMany Message
- `participants()` :belongsToMany User

---

#### Appointment (Rendez-vous confirmé)
**Responsabilités** :
- Gérer les rendez-vous confirmés
- Suivi des statuts et paiements finaux
- Gestion des litiges

**Champs clés** :
- `booking_request_id` : Lien vers la demande
- `bookable_id`, `bookable_type` : Polymorphique
- `status` : scheduled, completed, no_show, disputed, cancelled
- `final_price`, `deposit_amount` : Financier
- `dispute_*` : Gestion litiges

**Relations** :
- `bookingRequest()` :belongsTo BookingRequest
- `bookable()` :morphTo (artiste)
- `client()` :belongsTo Client

---

## 3. RELATIONS ENTRE MODÈLES
==============================================

### 🔄 Relations Polymorphiques (Core Architecture)

#### BookingRequest polymorphique
```php
// Un artiste peut être Tattooer, Pierceur, ou StudioArtist
BookingRequest::where('bookable_type', 'App\Models\Tattooer')
BookingRequest::where('bookable_type', 'App\Models\Pierceur')
BookingRequest::where('bookable_type', 'App\Models\StudioArtist')
```

#### Appointment polymorphique
```php
// Même structure que BookingRequest
Appointment::where('bookable_type', 'App\Models\Tattooer')
```

#### Media polymorphique
```php
// Tous les modèles avec HasMedia
User, Client, Tattooer, Pierceur, StudioArtist, BookingRequest, Message
```

### 🔗 Relations Principales

#### User → Profils (Polymorphique)
```php
// User.php
public function profile() {
    return match($this->role) {
        'client' => $this->hasOne(Client::class),
        'tattooer' => $this->hasOne(Tattooer::class),
        'pierceur' => $this->hasOne(Pierceur::class),
        'studio' => $this->hasOne(Studio::class),
        'studio_artist' => $this->hasOne(StudioArtist::class),
    };
}
```

#### Client → BookingRequests → Appointment
```php
Client::class
    ->hasMany(BookingRequest::class)
    ->hasMany(Appointment::class) // via booking_requests
    ->hasMany(TattooHistory::class)
```

#### Artistes → BookingRequests (Polymorphique)
```php
Tattooer::class
    ->morphMany(BookingRequest::class, 'bookable')
    ->morphMany(Appointment::class, 'bookable')
    ->morphMany(Availability::class, 'owner')
    ->morphMany(WorkingHour::class, 'owner')
```

#### Conversation → Messages → Participants
```php
Conversation::class
    ->hasMany(Message::class)
    ->belongsToMany(User::class, 'conversation_user')
        ->withPivot(['role', 'last_read_at', 'is_muted'])
```

#### Studio → Artists
```php
Studio::class
    ->hasMany(StudioArtist::class)
    ->hasManyThrough(Tattooer::class, StudioArtist::class)
```

### 📊 Relations de Support

#### Media Collections
```php
// Chaque modèle HasMedia définit ses collections
Tattooer::class
    ->addMediaCollection('avatar')      // unique
    ->addMediaCollection('portfolio')   // multiple
    ->addMediaCollection('drawings')    // multiple
    ->addMediaCollection('before_after') // multiple
```

#### Compliance & Consent
```php
Client::class
    ->hasOne(Consent::class)
    ->hasMany(ClientCareSheet::class)

Tattooer::class
    ->hasMany(ComplianceRecord::class)
```

## 5. LOGIQUE MÉTIER GÉNÉRALE
==============================================

### 🔄 Workflow Principal (Booking Request)

#### 1. Création demande (Client)
```php
// Client soumet une demande
BookingRequest::create([
    'client_id' => $client->id,
    'bookable_id' => $tattooer->id,
    'bookable_type' => 'App\Models\Tattooer',
    'status' => 'pending',
    'tattoo_size' => 'medium',
    'body_zone' => 'arm',
    'description' => 'Dragon japonais',
]);
```

#### 2. Acceptation par l'artiste
```php
// Artiste accepte avec modal complet
$bookingRequest->update([
    'status' => 'awaiting_deposit',
    'price_range_min' => 300,
    'price_range_max' => 500,
    'proposed_dates' => ['2024-03-15', '2024-03-22'],
    'total_deposit_amount' => 150,
    'client_payment_deadline' => now()->addDays(7),
    'chat_status' => 'open',
    'chat_closes_at' => now()->addDays(7),
]);

// Créer conversation
$conversation = Conversation::create([
    'booking_request_id' => $bookingRequest->id,
    'expiry_type' => 'deposit_pending',
    'expires_at' => now()->addDays(7),
]);
```

#### 3. Paiement acompte
```php
// Stripe webhook traite le paiement
$bookingRequest->update([
    'status' => 'deposit_paid',
    'deposit_paid_at' => now(),
]);

// Conversation devient permanente
$conversation->transitionToPermanentPhase();
```

#### 4. Envoi design et confirmation
```php
// Artiste envoie le design
$bookingRequest->update([
    'status' => 'design_sent',
    'design_sent_at' => now(),
]);

// Client confirme → création RDV
$appointment = Appointment::create([
    'booking_request_id' => $bookingRequest->id,
    'bookable_id' => $bookingRequest->bookable_id,
    'bookable_type' => $bookingRequest->bookable_type,
    'appointment_datetime' => $confirmedDate,
    'status' => 'scheduled',
]);
```

### 💬 Cycle de vie du Chat

#### Phase 1 : En attente acompte (7 jours max)
```php
$conversation->initializeDepositPendingPhase(7);

// Auto-expiration si pas de paiement
if ($conversation->shouldExpire()) {
    $conversation->markAsExpired();
    $bookingRequest->update(['status' => 'expired']);
}
```

#### Phase 2 : Permanent (après acompte)
```php
$conversation->transitionToPermanentPhase();
// Chat ne expire plus, accessible indéfiniment
```

#### Phase 3 : Post-RDV
```php
if ($artist->isOnFreePlan()) {
    // FREE : Suppression immédiate
    $conversation->update(['expires_at' => now()]);
} else {
    // PRO : Archivage permanent avec préservation images
    $conversation->archive();
    $conversation->preserveImagesOnly(); // Vers ClientCareSheet
}
```

### 💰 Gestion des Abonnements

#### Plans disponibles
```php
// FREE : Limitations
- Chat limité à 7 jours après acompte (point à régler chat ouvert jusqu'à jour du rdv après payment d'acompte, mais aucun archivage des image ou autre contenu)
- Suppression conversations post-RDV
- Portfolio limité (20 images)
- Pas de statistiques avancées

// PRO : Avantages
- Chat permanent (amettre en place - jusqu'à rdv passé (j+1))
- Archivage conversations avec images
- Portfolio illimité
- Statistiques avancées
- Export données
- Support prioritaire
- Suivit client (suivi des rendez-vous, consentements, traçabilité, etc.)
```

#### Vérification plan
```php
// Trait HasSubscription
public function isOnFreePlan(): bool {
    return $this->current_plan === 'free' || !$this->is_subscribed;
}

public function canKeepPermanentConversations(): bool {
    return $this->isOnProPlan();
}
```

```php
    'pending_requests' => BookingRequest::where('bookable_id', $tattooer->id)
        ->where('status', 'pending')->count(),
    'monthly_revenue' => TattooHistory::where('bookable_id', $tattooer->id)
        ->whereMonth('tattoo_date', now()->month)
        ->sum('total_paid'),
    'unread_messages' => $tattooer->getUnreadMessagesCount(),
];
```

### 🎯 Business Logic Summary

1. **Multi-rôles** : Architecture polymorphique flexible
2. **Workflow structuré** : Booking Request → Acompte → Chat → RDV
3. **Monétisation** : Abonnements FREE/PRO avec limitations
4. **Conformité** : Vérifications SIRET, consentements, blacklist
5. **Scalabilité** : Traits réutilisables, relations polymorphiques
6. **Sécurité** : Validation, autorisations, gestion mineurs
7. **Performance** : Soft deletes, indexes, caching possible

Cette architecture permet une évolution vers d'autres types de prestataires (coiffeurs, esthéticiennes, etc.) avec un minimum de changements grâce au système polymorphique et aux traits réutilisables.
